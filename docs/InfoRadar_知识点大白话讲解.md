## InfoRadar 知识点大白话讲解

配合《InfoRadar 模块说明与知识点手册》使用。手册告诉你"代码怎么写"，这份文档告诉你"为什么要这么写"和"这些东西到底是什么意思"。

---

### 先看全局：InfoRadar 到底在干嘛

想象你是一个情报机构的主管，手下有一支团队帮你盯梢各种话题——"上海 AI Agent 实习"、"LangChain 新版本动态"、"某某公司最新融资"。你不可能自己天天刷网页，所以你雇了一支"AI 情报团队"，每个人各司其职：

**Planner（情报经理）**：你告诉他要盯什么话题，他帮你想出 3-5 个搜索策略。比如"搜 LinkedIn 上这个方向的岗位"、"搜 36 氪的相关报道"、"搜 GitHub 上的新项目"。

**Researcher（情报侦察兵）**：拿着 Planner 给的搜索策略，真的去互联网上搜。搜回来一堆结果后，他还会给每条结果打分——"这条跟话题高度相关给 5 分"、"这条是广告给 1 分"。低分的直接扔掉，高分的才存进档案库。

**Analyst（情报分析师）**：拿到 Researcher 收集到的新信息后，先看看有没有什么变化——"这次比上次多了 5 条字节跳动的岗位"、"某某框架发布了新版本"。如果没啥新东西，他就不干活（省钱）。有变化的话，他再写一段分析总结。

**Writer（情报简报撰稿人）**：拿到 Analyst 的分析结果，润色成一份漂亮的情报简报，发给老板（你）看。

**Archiver（档案管理员）**：数据攒太多了，30 天前的热乎数据已经不那么热了。他把旧数据压缩成摘要归档，但原文不删——万一以后要翻旧账呢。

这五个人组成一条流水线：Planner → Researcher → Analyst → Writer，Archiver 独立运行。这就是 InfoRadar 的核心架构。

---

### 第 1 周：基础骨架 + 搜索能力

第 1 周的目标很简单：让系统能搜到东西。先把地基打好，把 Planner 和 Researcher 跑通。

---

#### 1.1 `.env` — 环境变量：密码不能写在代码里

**大白话**：你的 API 密钥就像银行卡密码，不能写在程序代码里然后传到 GitHub 上去。`.env` 文件就是一个专门的"密码本"，程序启动的时候去这个文件里找密码，但这个文件不会被上传到 Git 里（被 `.gitignore` 排除了）。

**生活类比**：你在公司填报销单，单子上写着"请财务部核对银行账号"，但你的银行账号写在另一张保密表格里，不会贴在报销单上让所有人看到。`.env` 就是那张保密表格。

**12-Factor App 是什么**：这是一套做 Web 应用的最佳实践（总共 12 条规则），其中第三条就是"配置存储于环境中"。意思是：所有会变的东西（数据库地址、API Key、模型名称）都放在环境变量里，不要硬编码在代码里。这样你换个环境（比如从自己的电脑部署到服务器上），只需要改 `.env` 文件，不用改任何代码。面试的时候你可以说"我们的项目遵循 12-Factor App 原则，配置外置于环境变量"，面试官一听就知道你懂工程规范。

**Tavily 是什么**：一个专门为 AI Agent 设计的搜索 API。你可能会想"为什么不直接用 Google 搜索？"——因为 Google 搜索返回的是 HTML 网页，你得自己解析，而且 Google 有反爬虫机制，程序化调用经常被封。Tavily 专门做了优化，返回的结果是结构化的（标题、URL、摘要都是分开的字段），对 AI 很友好。就像你去餐厅吃饭（用 Tavily）和自己买菜回家做饭（爬 Google）——结果都能吃饱，但前者省时间。

---

#### 1.2 `requirements.txt` — 依赖列表：买菜清单

**大白话**：你的程序不是一个文件就能跑的，它需要很多"外援"——第三方库。`requirements.txt` 就是一份"买菜清单"，列出了这个项目需要哪些依赖包以及最低版本号。别人拿到你的代码后，跑一句 `pip install -r requirements.txt` 就能把所有需要的包都装好。

**几个关键依赖是干嘛的**：

`langgraph`：LangChain 团队出的 Agent 编排框架，用来把多个 Agent 串成流水线。后面会详细讲。

`langchain-openai`：虽然名字里有"openai"，但它其实能对接任何兼容 OpenAI 接口的 LLM——包括 DeepSeek。你只需要把 API 地址换成 DeepSeek 的就行。这就像一个万能插头，只要插座标准一样，什么电器都能插。

`celery` + `redis`：Celery 是做定时任务的（"每天早上 9 点自动扫描"），Redis 是 Celery 的"消息中转站"。后面会详细讲。

`chromadb`：一个嵌入式的向量数据库，用来存网页正文的"语义信息"。后面会详细讲。

`sse-starlette`：做实时推送的。扫描的时候前端能实时看到 Agent 的执行进度，靠的就是它。

**为什么 pydantic-settings 要单独装**：在旧版本的 pydantic 里，`BaseSettings` 是内置的。但 pydantic v2 把它拆成了独立包，所以需要单独 `pip install pydantic-settings`。这是很多人会踩的坑。

---

#### 1.3 `app/__init__.py` — 包标记：门牌号

**大白话**：Python 需要知道一个文件夹是"包"（package）才能用 `from app.xxx import yyy` 这种语法导入里面的模块。`__init__.py` 就是告诉 Python "这个文件夹是个包"的门牌号。它可以是空的，但必须存在。

**生活类比**：一栋写字楼里有很多办公室，但如果没有门牌号，快递就找不到地方。`__init__.py` 就是那个门牌号。

---

#### 1.4 `app/config.py` — 配置管理：一个聪明的管家

**大白话**：你的程序有很多配置项——API Key 放在哪、数据库文件存在哪、多少天前的数据要归档、最低分数线是多少。如果把这些配置散落在各个文件里，改一个设置要翻好几个文件。`config.py` 把所有配置集中到一个地方，程序启动时自动从 `.env` 文件读取。

**pydantic-settings 做了什么**：它不只是读取 `.env`，还会做类型校验。比如 `archive_after_days` 声明是 `int` 类型，如果你在 `.env` 里写了 `archive_after_days=abc`，程序启动时直接报错，而不是等到运行时才发现数据类型不对。这就像你填表格时，"年龄"那一栏只能填数字，填了字母系统会提示你格式错误。

**单例模式是什么意思**：`get_settings()` 函数用了一个全局变量 `_settings`。第一次调用时，它创建 `Settings()` 实例并保存；之后再调用，直接返回之前创建的实例，不会重新创建。整个程序生命周期里只有一个 Settings 实例。

为什么要这样？两个原因：一是避免浪费（每次调用都重新读 `.env` 文件、重新校验类型，没必要）；二是保证一致性（全程序用同一份配置，不会出现"这个文件读到的是新配置，那个文件读到的还是旧配置"的问题）。

**生活类比**：公司里只有一个财务总监（单例）。不管哪个部门要报销，都找同一个人。不会出现"A 部门找张三报销、B 部门找李四报销、两个人标准不一样"的情况。

---

#### 1.5 `app/models.py` — 数据模型：信息的模板

**大白话**：这个文件定义了程序里用到的四种数据结构。你可以把它们想象成四种"表格模板"。

**Topic（话题模板）**：用户在 Web UI 上添加一个追踪话题时，系统就创建一个 Topic 对象。里面有话题名字（"上海 AI Agent 实习"）、扫描频率（每 24 小时一次）、创建时间等。

**Finding（信息发现模板）**：每次搜索到一条有价值的信息，就创建一个 Finding。里面有 URL、标题、公司名、相关度评分（1-5 分）、冷热标记（hot/cold/archive）等。

**@dataclass 是什么**：Python 的 `@dataclass` 装饰器帮你自动生成 `__init__`、`__repr__` 等方法。如果你手写一个 Topic 类，你需要写 `def __init__(self, id=0, name="", ...)` 一大坨。加上 `@dataclass` 之后，只要声明字段和类型，Python 自动帮你生成。

**生活类比**：你在 Excel 里创建表格，`@dataclass` 就像 Excel 的"模板"功能——你定义好列名和数据类型，每新增一行数据都按这个格式来，不用每次手动对齐。

**TypedDict 是什么**：普通的 Python 字典 `dict` 是什么都能装，但你不知道里面有什么 key、每个 value 是什么类型。`TypedDict` 给字典加了"模板"——它声明了"这个字典必须有这些 key，每个 key 对应什么类型"。LangGraph 用 `RadarState`（TypedDict）来校验状态结构，如果你写错了 key 名，它会报错。

**RadarState（流水线共享状态）**：这是 LangGraph 最核心的概念之一。四个 Agent 在流水线上工作时，它们共享一个"状态包"。Planner 往里面放搜索策略，Researcher 从里面取策略、往里面放搜索结果，Analyst 从里面取结果、往里面放分析……就像一条流水线上放了一个共享的篮子，每个工人往里放东西或从里面取东西。

**tier 三态是什么意思**：每条 Finding 有一个 `tier` 字段，标记它是"热区"（hot）、"冷区"（cold）还是"归档区"（archive）。刚搜到的都是 hot（热乎的、随时可能被检索），30 天后降级为 cold（不太热了、低频检索），归档后摘要存到 archive 区。这就像你手机里的照片——最近拍的随时翻看（hot），几个月前的偶尔翻翻（cold），几年前的已经存到云端了平时不看（archive）。

**content_hash 为什么需要**：同一条信息可能从不同的 URL 搜到（比如同一个新闻被两个网站转载了）。`content_hash` 是对正文内容算一个 MD5 哈希值——如果两篇文章的正文一样，它们的 hash 也一样，系统就知道"这条已经存过了，跳过"。就像快递的运单号，不管从哪个渠道查询，同一个包裹的单号是一样的。

#### 1.6 `app/tools.py` — 工具函数集：所有 Agent 共用的工具箱

**大白话**：四个 Agent 都要用到一些"底层能力"——搜索互联网、读写数据库、抓取网页。这些能力不是某个 Agent 独有的，而是所有 Agent 共用的。`tools.py` 就是存放这些公共工具的地方。

**单例客户端（get_tavily / get_chroma / get_db）**：和 `get_settings()` 一样的思路。Tavily 客户端、ChromaDB 客户端、SQLite 连接，每个只创建一次，全局复用。如果每次搜索都重新创建一个 Tavily 客户端，就需要重新建立网络连接，既慢又浪费资源。

**生活类比**：你家的水龙头（客户端）是一直装在那里的。你每次用水，拧开就行（`get_tavily()`），不用每次都重新接一根水管（重新创建客户端）。

**init_db() — 建表**：SQLite 里建三张表——`topics`（话题配置）、`findings`（信息发现）、`digests`（情报摘要）。用的是 `CREATE TABLE IF NOT EXISTS`，意思是"如果表不存在就创建，存在就跳过"。这样你程序启动多次也不会重复建表或清空数据。

三张表的关系：`topics` 和 `findings` 通过 `topic_id` 关联——一个话题下可以有多条 Finding。`topics` 和 `digests` 也是通过 `topic_id` 关联——一个话题可以有多份摘要。

**混合存储（SQLite + ChromaDB）**：这是 InfoRadar 的一个重要设计决策。

SQLite 负责存"结构化数据"——URL、标题、公司名、评分、时间。这些数据有明确的字段，查询方式也很精确："帮我找所有公司名是'字节跳动'的 Finding"、"这个 URL 存过了吗"。

ChromaDB 负责存"语义数据"——网页正文。ChromaDB 会把文本转成向量（一串数字），之后你可以用"语义相似"来搜索。比如你搜"AI 工程师实习"，它能找到正文里写了"机器学习实习生"的文章——虽然关键词不完全一样，但语义是相近的。

为什么要分开？因为两种数据需要不同的查询方式。结构化数据需要精确匹配（SQL 查询），语义数据需要模糊匹配（向量搜索）。一个数据库很难同时做好两件事，所以用两个各取所长。

**生活类比**：你的通讯录就是"结构化数据"（姓名、电话、地址都是固定字段，精确查找）。你的读书笔记就是"语义数据"（内容是自由文本，你想搜"之前看过的关于时间管理的那段话"就需要模糊搜索）。你不会把通讯录和读书笔记存在同一个软件里。

**web_search — 搜索互联网**：调用 Tavily API 搜索，返回结构化的结果列表（URL、标题、摘要、相关性分数）。Tavily 自带摘要功能，不需要你再去抓网页就能拿到一段内容概要。

**fetch_page — 抓取网页正文**：用 httpx 发送 HTTP 请求获取网页 HTML，然后用 BeautifulSoup 去掉 `<script>` 和 `<style>` 标签（那些是广告和样式，不是正文），提取纯文本。截断到 8000 字符，避免存太多不重要的内容。

**url_exists — URL 去重**：查 SQLite 看看这个 URL 是不是已经存过了。存过了就跳过，不重复入库。

**content_hash — 内容哈希**：用 MD5 对文本算哈希值。两段相同的文本会得到相同的哈希值。在 `save_finding` 之前算好 hash 存进去，之后可以用来判断内容是否重复。

**detect_changes — 规则引擎**：这个函数做的事情很"确定性"——哪些 URL 是新的（和数据库里已有的 URL 对比）、每家公司出现了几次（频率统计）、标题里有没有版本号（正则提取 `v2.1.3` 这种）。这些事用代码做比用 LLM 做更好：零 token 消耗、速度快、结果 100% 准确。LLM 做这种计数反而可能出错。

**生活类比**：你去超市清点货架，数"可口可乐有几瓶、百事可乐有几瓶"——这种事用眼睛数就行（代码做），不需要请一个博士来帮你数（LLM 做）。

#### 1.7 `app/planner.py` — Planner Agent：情报经理

**大白话**：Planner 是流水线的第一个 Agent。它的工作很简单：你告诉它要盯什么话题（"上海 AI Agent 实习"），它调用一次 LLM，生成 3-5 个搜索查询。比如它可能会生成："LinkedIn 上海 AI Agent 实习岗位"、"36 氪 Agent 创业公司融资"、"GitHub awesome AI agent intern"。

**SystemMessage 是什么**：LLM 收到的消息分两种——系统消息（SystemMessage）和用户消息（HumanMessage）。系统消息是给 LLM 的"角色设定"，告诉它"你是一个情报分析经理，只输出搜索查询，不要闲聊"。用户消息才是真正的输入："话题：上海 AI Agent 实习"。

**生活类比**：SystemMessage 就像你给新员工发的岗位说明书——"你的岗位是情报经理，职责是生成搜索策略，不要做别的事"。HumanMessage 就像你给他的具体任务——"帮我查一下这个方向的信息"。

**输出解析为什么要做**：LLM 有时候很"话多"。你让它输出搜索查询列表，它可能输出：

```
1. LinkedIn 上海 AI Agent 实习
2. 36 氪 Agent 创业公司
3. GitHub awesome agent intern
```

前面带了编号。Planner 的代码会把编号去掉：检测到行首是数字，就用 `split(".", 1)` 把 "1. " 去掉，只保留后面的查询文本。

**temperature=0.3 是什么意思**：temperature 控制 LLM 输出的"随机性"。0 表示完全确定（同样的输入永远得到同样的输出），1 表示非常有创造性（每次输出都不一样）。0.3 是一个适中的值——搜索策略需要一定的多样性（不想每次都搜一样的东西），但也不能太放飞自我（搜出来的东西得跟话题相关）。

---

#### 1.8 `app/researcher.py` — Researcher Agent：情报侦察兵

**大白话**：Researcher 是干脏活累活的。它拿到 Planner 给的搜索查询，真的去互联网上搜，搜回来一堆结果后，打分、过滤、存储。

**整个流程分四步**：

第一步：**搜索**。拿着 Planner 给的 3-5 个查询，每个搜 5 条结果，全部合并到一起。

第二步：**去重**。不同查询可能搜到同一条结果（URL 相同）。用一个 `set` 做去重——`not seen.add(r["url"])` 这行代码很巧妙，`set.add()` 返回 `None`，`not None` 是 `True`，所以第一次遇到的 URL 保留，重复的过滤掉。

第三步：**LLM 打分**。把搜索结果打包发给 LLM，让它给每条结果打 1-5 分。这里有个重要的设计：不逐条打分，而是把所有结果一起发给 LLM 一次性评完。为什么？因为 LLM 调用是按 token 计费的。10 条结果打包发一次，只算一次调用的 token；如果逐条打分，就是 10 次调用，token 消耗翻 10 倍。

**生活类比**：你要批改 30 份试卷。逐份批改——每改完一份就站起来走一圈再坐下来改下一份（10 次 LLM 调用）。打包批改——30 份摊在桌上一次性全改完（1 次 LLM 调用）。显然打包更高效。

第四步：**过滤 + 存储**。低于 3 分的扔掉（不相关），URL 已经存过的跳过（去重），4-5 分的高分结果才去抓全文（分级抓取，控制成本），3 分的只存 Tavily 自带的摘要。最后把通过的 Finding 同时写入 SQLite 和 ChromaDB。

**为什么要分级抓取**：抓全文是要发 HTTP 请求的，慢且可能被目标网站限制。不是所有结果都值得花这个代价——3 分的"还行"结果存个摘要就够了，只有 4-5 分的"很好"结果才值得花时间去抓全文。

### 第 2 周：完整流水线

第 1 周让系统"能搜到东西"，第 2 周把剩下的 Agent 补齐，串成完整流水线。

---

#### 2.1 `app/analyst.py` — Analyst Agent：情报分析师（核心差异化）

**大白话**：Analyst 是 InfoRadar 和普通"搜索+摘要"工具的核心区别所在。它不是拿到一堆数据就直接让 LLM 写总结，而是分两阶段：先让代码做确定性分析，再让 LLM 做"需要理解力"的总结。

**两阶段分析**：

阶段一（规则检测）：调用 `detect_changes()`（第 1 周写的那个规则引擎），看看这次搜索和上次比有什么变化——新增了几条？哪些公司出现频率变高了？有没有新版本发布？这些操作用纯代码做，零 token 消耗，又快又准。

阶段二（LLM 总结）：如果阶段一发现了变化，把变化的结构化数据（JSON 格式）喂给 LLM，让它写一段"人话"总结。LLM 看到的不是几百条原始搜索结果，而是已经提炼过的"新增 3 条、字节频率上升"这样的信息。

**为什么要两阶段**：如果直接把几百条搜索结果扔给 LLM 让它"分析一下有什么变化"，它可能数错（LLM 的计数能力很差），而且 token 消耗巨大。先用代码做计数和对比（零成本、100% 准确），再把结论给 LLM 做"翻译"（把数据变成人话），这才是最优解。

**生活类比**：公司季度财报出来了。你不会把几千行 Excel 数据直接扔给 CEO 说"你看看有什么变化"。你会先让财务部门整理成"收入增长 15%、成本下降 8%"（代码做确定性分析），再让分析师写成一段"本季度营收增长主要由海外业务驱动……"（LLM 做趋势归纳）。

**条件跳过（没变化不调 LLM）**：如果规则检测发现"没有新增条目、没有频率变化、没有版本变化"——那就没必要调 LLM 了，直接返回"本次扫描未发现明显变化"。这省了一次 LLM 调用，也就省了 token 费用。

**json.dumps 的 ensure_ascii=False**：Python 的 `json.dumps` 默认会把非 ASCII 字符转成 `\uXXXX` 格式。也就是说 "字节跳动" 会变成 `"\u5b57\u8282\u8df3\u52a8"`。加了 `ensure_ascii=False` 之后，中文保持原样。

---

#### 2.2 `app/writer.py` — Writer Agent：情报简报撰稿人

**大白话**：Writer 是整个流水线里最简单的 Agent——它就是把 Analyst 的分析结果润色成一份漂亮的 Markdown 格式摘要。本质上就是一次 LLM 调用。

**结构化 Prompt**：Writer 的 SystemMessage 里规定了输出格式——"## 本期要点"、"## 新增信息"、"## 趋势观察"、"## 来源"。这样 LLM 输出的就是标准格式，直接渲染到网页上就能看，不需要再做格式转换。

**生活类比**：你给秘书一个任务："把这些数据写成一份周报，格式是：本周要点 3-5 条，新增事项，趋势分析，信息来源。"秘书照着这个格式写就行了。

---

#### 2.3 `app/archiver.py` — Archiver Agent：档案管理员

**大白话**：数据攒多了会越来越多，全部放在"热区"既浪费空间又影响检索速度。Archiver 的工作就是定期（每周日凌晨 3 点）把旧数据归档——30 天前的热区数据降级为冷区，同时生成压缩摘要。

**冷热分层三态**：

hot（热区）：刚搜到的数据，全文可检索。就像你桌上正在看的文件。

cold（冷区）：30 天前的数据，原文还在，但不在日常检索范围内。就像你把不常看的文件放到了柜子里。

archive（归档区）：LLM 压缩摘要后的数据，原文已被"冷处理"。就像你把一整年的文件浓缩成一页纸的年度总结。

**为什么不删原文**：删了就没有回头路了。万一以后想做更细粒度的分析，或者发现之前某条被忽视的信息现在变得重要了，冷区的原文还在可以找回来。归档的目的是"让日常检索更快"，不是"清理硬盘"。

**生活类比**：你的手机照片管理。最近一个月的照片在"最近项目"里随时翻看（hot），三个月前的照片还在相册里但不常翻（cold），你把某一年的所有照片整理成了一本精选相册（archive），精选相册里只有最好的几张，但原始照片全在 iCloud 里没有删。

**按周聚合**：Archiver 不是逐条生成摘要（那样 token 消耗太大），而是把同一周的所有条目打包在一起，让 LLM 生成一段综合摘要。比如一周有 50 条招聘信息，压成一段"本周字节跳动和百度分别发布了 3 个 Agent 相关岗位，薪资范围 200-400/天"。压缩比大约 5:1。

#### 2.4 `app/graph.py` — LangGraph 状态图：流水线指挥官

**大白话**：前面四个 Agent（Planner、Researcher、Analyst、Writer）各自是独立的函数。`graph.py` 的工作就是用 LangGraph 把它们串成一条流水线，并且管理好它们之间的数据传递。

**StateGraph 是什么**：LangGraph 提供的核心数据结构。你定义"节点"（每个节点是一个 Agent 函数）和"边"（节点之间的连接关系），LangGraph 就知道怎么按顺序执行它们。

```
START → planner → researcher → analyst → writer → END
```

就像工厂的流水线：原料从 START 进入，经过四个工位（节点），每个工位加工完把半成品放到传送带（state）上，下一个工位从传送带上取走继续加工，最后成品从 END 出来。

**节点返回 dict 是什么意思**：每个节点函数接收完整的 state（篮子里所有的东西），做完工作后只返回它需要更新的字段。LangGraph 自动把这些更新合并到 state 里。

比如 Planner 只返回 `{"search_queries": ["查询1", "查询2", ...]}`，LangGraph 就把 `search_queries` 这个字段更新到 state 里，其他字段不变。下一个节点 Researcher 从 state 里取出 `search_queries`，做完后返回 `{"findings": [...], "sources": [...]}`，LangGraph 再把这两个字段更新进去。

**生活类比**：四个厨师做一道菜。第一个厨师切好菜放到案板上（返回 search_queries），第二个厨师从案板上取菜炒好放到盘子里（返回 findings），第三个厨师在盘子旁边加上装饰（返回 analysis），第四个厨师最后摆盘上桌（返回 digest）。每个人只动自己负责的部分，不会把别人做好的东西弄乱。

**compile() 复用**：图编译一次后就不可变了（不能中途加节点或改边）。所有话题的扫描都复用同一个编译好的图。就像工厂的流水线搭建好后，不管是生产手机还是生产平板，用的都是同一条流水线——只是放进去的原料（topic）不一样。

**线性流水线 vs 循环**：InfoRadar 的流水线是直线的，没有循环——Planner 做完就给 Researcher，不会回头。这和 CodeGuard 不同（CodeGuard 用 ReAct 循环，Agent 可以反复思考和行动）。为什么 InfoRadar 不需要循环？因为情报监控是一次性扫描——搜完了、分析完了、出摘要了，就结束了。不需要"分析师觉得不够好，让侦察兵再搜一轮"。

### 第 3 周：定时调度 + Web UI

第 1-2 周做的是"手动触发扫描"。第 3 周加上自动化——Celery Beat 定时触发，FastAPI 提供 Web API，用户通过网页管理话题和查看结果。

---

#### 3.1 `app/tasks.py` — Celery 任务 + Beat 调度：自动闹钟

**大白话**：你不可能每天手动去点"开始扫描"。Celery Beat 就是一个自动闹钟——你设定好"每小时检查哪些话题到期了"、"每周日凌晨 3 点归档旧数据"，它就会自动执行。

**Celery 的三个角色**：

Beat（调度器）：负责看时间，到点了就往队列里丢一个任务消息。它自己不做任何实际工作，只负责"喊人"。

Redis（消息队列）：Beat 把任务消息丢到 Redis 里，Worker 从 Redis 里取任务来做。Redis 就像一个公告板——Beat 把"请扫描话题 A"贴在公告板上，Worker 看到就去执行。

Worker（干活的）：从 Redis 里取任务，真正执行扫描或归档。

**为什么要用 Celery 而不是 Python 的 sleep 循环**：`sleep` 循环会让你的程序一直占着一个进程。Celery 的好处是解耦——Beat 和 Worker 是独立的进程，Worker 崩溃了不影响 Beat 继续定时，重启 Worker 就能接着干活。

**crontab 表达式**：一种配置定时任务的语法。`crontab(minute=0)` 表示"每小时的第 0 分钟"（也就是每小时整点）。`crontab(hour=3, minute=0, day_of_week=0)` 表示"每周日凌晨 3:00"。

**task_acks_late 是什么意思**：ack 是 acknowledgment（确认）的缩写。默认情况下，Worker 收到任务后就立刻 ack（"我收到了"），然后开始执行。如果执行到一半 Worker 崩溃了，这个任务就丢了（因为已经 ack 过了）。设置 `task_acks_late=True` 后，Worker 执行完才 ack。如果中途崩溃，任务还没 ack，会被重新放回队列让别的 Worker 来做。

**生活类比**：快递签收。默认模式是"快递员放到驿站就算送达了"（task_acks_early），驿站倒闭了你的快递就丢了。late 模式是"你亲手签收了才算送达"（task_acks_late），你没签收的话快递员会把包裹带回去明天再送。

**.delay() 是什么意思**：`scan_topic.delay(id)` 不是立刻执行 `scan_topic`，而是把一个任务消息异步发送到 Redis 队列里，Worker 会从队列里取出来执行。调用 `.delay()` 的代码不会等待结果，立刻就能继续执行下一个任务。这就是"异步"的含义。

**include=["app.tasks"]**：这行看起来不起眼，但没有它 Celery Worker 就找不到你定义的任务，会报 `NotRegistered` 错误。它告诉 Celery "去 app.tasks 这个模块里找任务定义"。这是从 CodeGuard 项目踩坑学来的经验。

**Redis 用 /1 和 /2**：Redis 默认有 16 个数据库（0-15）。Celery 的 broker（消息队列）用 `/1`，backend（结果存储）用 `/2`，避免消息和结果混在一起。就像你把"待办事项"和"已完成事项"放在两个不同的文件夹里。

---

#### 3.2 FastAPI 入口 — API 就像餐厅的服务员

**大白话**：你的程序核心功能（扫描、存储、分析）都在 Python 代码里。但用户不可能打开终端来操作。FastAPI 把这些功能包装成 HTTP API（接口），用户通过网页或手机发 HTTP 请求就能使用。

FastAPI 的角色就像餐厅的服务员——顾客（浏览器）说"我要一份宫保鸡丁"（POST /api/scans/1），服务员把菜单传给后厨（Python 代码），后厨做好了，服务员再把菜端上来（返回 JSON 结果）。

---

#### 3.3 SSE 实时推送 — 像直播一样看扫描进度

**大白话**：一次完整的扫描可能需要 10-30 秒。如果用户在网页上点"扫描"后只看到一个转圈圈，体验很差。SSE（Server-Sent Events）让服务器可以实时向前端推送消息——Planner 开始工作了，前端显示"规划搜索策略中..."；Researcher 搜完了，前端更新"搜索完成，发现 12 条结果"；Writer 写完了，前端显示最终的摘要。

**SSE vs WebSocket**：

WebSocket 是双向的——服务器可以推消息给浏览器，浏览器也可以推消息给服务器。就像打电话，双方都能说话。

SSE 是单向的——只有服务器推消息给浏览器。就像听广播，你只能听不能说话。

InfoRadar 只需要单向推送（服务器告诉前端扫描进度），所以用 SSE 就够了，比 WebSocket 更简单。

### 第 5 周：React + Vite 前端

第 3 周的 Web UI 是单文件 HTML，够简单但不好维护。第 5 周用 React + Vite + Tailwind CSS 搭一个正经的前端项目。

---

#### 5.1 Vite — 比 Webpack 快 10 倍的构建工具

**大白话**：React 写的 JSX 代码浏览器不能直接运行，需要"构建工具"把它转成浏览器能识别的 JavaScript。Vite 就是这样一个工具，它的特点是极快——开发时修改代码，页面几乎瞬间刷新（热更新）。

**Vite proxy 做了什么**：开发时前端跑在 `localhost:5173`，后端跑在 `localhost:8000`，端口不同，浏览器会因为安全策略阻止跨域请求。Vite 的 proxy 配置让所有 `/api` 开头的请求自动转发到后端，浏览器以为是同一个服务器，就不会阻止了。

**生活类比**：你在 A 商场（前端 5173 端口）想买 B 商场（后端 8000 端口）的东西，正常情况保安不让你跨商场购物。proxy 就像 A 商场里设了一个 B 商场的代购点——你在 A 商场里下单，代购点帮你去 B 商场拿货，你觉得一直在 A 商场。

---

#### 5.2 Tailwind CSS — 不用写 CSS 文件的样式框架

**大白话**：传统的做法是写一个 `.css` 文件，里面定义 `.button { background: blue; color: white; }`，然后在 HTML 里用 `className="button"` 引用。Tailwind 的做法是直接在 HTML 里写 `className="bg-blue-500 text-white"`——样式就写在 class 里，不需要单独的 CSS 文件。

**@theme 自定义颜色**：Tailwind 默认的颜色不一定符合你的设计需求。`@theme` 让你定义自己的颜色变量——`--color-primary: #3B82F6` 定义了主色，之后在组件里用 `bg-primary` 就是蓝色，`text-primary` 也是蓝色。改一个地方，所有用到主色的组件一起变。

---

#### 5.3 React 核心概念

**useState — 组件的记忆**：React 组件默认是"无状态"的——每次渲染都从零开始。`useState` 让组件能"记住"一些数据。比如 `const [topics, setTopics] = useState([])` 让组件记住话题列表，`setTopics(newList)` 更新列表时页面自动刷新。

**useEffect — 副作用执行器**：组件加载后要做一些事情（比如发 API 请求加载数据），`useEffect` 就是干这个的。它在组件渲染完成后执行你指定的函数。

`useEffect(() => { loadData() }, [])` 的 `[]` 表示"只在组件第一次加载时执行一次"。如果改成 `[id]`，表示"id 变化时重新执行"。

**useEffect 清理函数**：`return () => esRef.current?.close()` 这行是清理函数。当组件卸载时（比如用户切换到另一个页面），React 会自动调用这个清理函数来关闭 SSE 连接。如果不做清理，旧的 SSE 连接会一直开着，浪费资源，这叫"内存泄漏"。

**生活类比**：你在酒店退房时，清洁人员会自动来打扫房间（清理函数）。如果你不辞而别（没有清理函数），房间就一直占着（内存泄漏）。

**useRef — 不触发重渲染的存储**：`useState` 更新数据时页面会重新渲染。但有些数据更新了不需要页面刷新——比如 EventSource 实例（SSE 连接对象）。`useRef` 就是存放这种数据的地方。

**useParams — 获取 URL 参数**：`/topic/:id` 路由中的 `:id` 是动态的。`useParams()` 返回 `{ id: "1" }`，组件就知道当前是哪个话题。

**Promise.all — 并行请求**：`await Promise.all([getTopics(), getStats()])` 同时发两个请求，等两个都回来再继续。如果分开写 `await getTopics()` 再 `await getStats()`，第二个请求要等第一个回来才开始，多浪费一半时间。

**生活类比**：你去银行办事，同时需要取号和填表。Promise.all 就是你一边等叫号一边填表（并行），而不是先等叫到号再开始填表（串行）。

**key 强制重渲染**：`<ScanPanel key={scanKey} />` 的 `scanKey` 每次点"扫描"都加 1。React 发现 key 变了，就会销毁旧的 ScanPanel 组件、创建一个新的。新组件的 `useEffect` 会重新执行，SSE 连接重新开始。这比在组件内部手动管理"关闭旧连接、建立新连接"更干净。

---

#### 5.4 CORS — 跨域资源共享

**大白话**：浏览器的安全策略规定，网页只能访问同域（同协议+同域名+同端口）的资源。前端在 `localhost:5173`，后端在 `localhost:8000`，端口不同，算"跨域"。浏览器默认会阻止跨域请求。

CORS 中间件就是在后端告诉浏览器："我允许 `localhost:5173` 来访问我的 API，你别拦着。"

**生活类比**：你家小区门禁默认只让本小区住户进入（同源策略）。外卖小哥（前端请求）进不来，除非物业（CORS 中间件）给外卖小哥登记放行。

---

#### 5.5 EventSource — 浏览器原生 SSE 客户端

**大白话**：`new EventSource(url)` 是浏览器内置的 SSE 客户端，不需要装任何第三方库。它和服务器建立一个长连接，服务器可以随时通过这个连接推送事件给浏览器。

`addEventListener('progress', ...)` 监听"progress"事件——后端推送一个 progress 事件，前端就更新 UI 显示"Researcher 正在搜索..."。

`addEventListener('complete', ...)` 监听"complete"事件——扫描完成，前端显示最终摘要。

**可选链 ?.()**：`onProgress?.(data)` 等价于 `if (onProgress) { onProgress(data) }`。如果 `onProgress` 是 `undefined`（没传这个回调），`?.()` 安全地跳过，不报错。

### 知识点之间的联系（面试加分项）

面试时单独讲每个知识点不难，难的是把它们串起来讲。以下是几个串联角度：

**"你的项目和 ChatGPT 有什么区别？"**

ChatGPT 是一次性对话。InfoRadar 是持续监控——Celery Beat 定时触发，每轮扫描是完整的 4 Agent 流水线，结果存入知识库积累。知识库不是平面的——有混合存储（SQLite + ChromaDB）和冷热分层（hot/cold/archive），让数据越攒越有价值而不是越攒越乱。

**"为什么用 LangGraph 而不是 AutoGen/CrewAI？"**

InfoRadar 的 4 Agent 之间是确定性的线性流转——数据分析完了就交给 Writer，不需要 Agent 之间讨论或投票。LangGraph 的 StateGraph 完美匹配这个模式。AutoGen 擅长的是 Agent 之间的多轮对话和协作推理，用在这里是杀鸡用牛刀。

**"怎么控制 token 成本？"**

三道关卡。第一道：Researcher 的批量打分（10 条结果 1 次调用，不是 10 次）。第二道：Analyst 的条件跳过（没变化就不调 LLM）。第三道：规则引擎做确定性任务（URL 去重、频率统计、版本号提取全用代码，零 token）。

**"数据库为什么不全用 ChromaDB？"**

ChromaDB 擅长语义搜索（"找意思相近的内容"），但不擅长精确查询。"这个 URL 存过了吗"是精确匹配，用 SQL 一句话搞定；"字节跳动出现了几次"是聚合统计，ChromaDB 做不了。混合存储各取所长。
